= Documentation map

Where everything lives. Two homes, one rule:

* *The repo* holds *reference* — the durable truth: what it is, why it's built this way,
  and the config that proves it. Versioned with the code it describes.
* *The wiki* holds *help* — task-first, audience-first, safe to reword often. Its source
  lives in `wiki/` in this repo and is published to the GitHub wiki, so it's reviewed and
  versioned like everything else (see <<wiki-source>>).

Legend: ✅ done · ⚠️ needs an update · ▢ planned.

== Wiki source of truth — `wiki/`, not the web UI

A GitHub wiki is a *separate git repo*: outside this repo's review, history, and backups,
and editable out-of-band. For a *security* appliance that's the wrong default — a wiki page
is exactly where someone would slip in a malicious `curl … | sh`.

So *`wiki/` in this repo is the source of truth* and `bin/wiki-sync.sh` publishes it. Edits
go through a normal PR. `Home` says so, and the wiki should be set to *restrict editing to
collaborators* (Settings → Features → Wikis).

[source,sh]
----
sh bin/wiki-sync.sh --check   # diff repo vs published wiki, change nothing
sh bin/wiki-sync.sh           # publish wiki/ to the GitHub wiki
----

== Reference convention (what `bin/check-doc-links.sh` enforces)

[cols="1,1,2"]
|===
|Kind |Resolved from |Example

|Clickable link — `+[text](path)+` / `+link:path[]+`
|*the file it sits in* — that's what GitHub follows
|`+[.github/GOVERNANCE.md](../../.github/GOVERNANCE.md)+`

|Prose mention — `+`<dir>/<file>.md`+`
|*the repo root*
|`+`docs/HARDENING.md`+`
|===

Root-relative prose survives the *referring* file being moved; `../` in prose doesn't —
which is precisely what broke when these docs moved into `docs/`. Run the checker before
committing any move:

[source,sh]
----
sh bin/check-doc-links.sh   # exit 1 if any reference is dead
----
